# GeoProphet 平台原理、优缺点总结

生成日期：2026-06-06

## 一、平台定位

GeoProphet 是一个面向地质灾害监测、预警、分析和公众上报的 Web 平台 MVP。当前版本重点覆盖演示和联调场景，已经形成了“监测数据展示、地灾图层可视化、预警管理、群众上报、AI 图片初审、系统审计”的基本闭环。

平台适合用于项目汇报、业务原型验证、接口联调和后续真实模型/真实数据接入前的底座验证。若用于正式生产，还需要继续补齐实时数据接入、模型服务上线、任务队列、生产数据库治理、运维监控和账号体系。

## 二、总体架构原理

平台采用前后端分离加独立模型服务的三段式架构：

1. 前端展示层

   前端使用 Vue 3、Vite、Element Plus、Pinia、Vue Router、ECharts 和 Leaflet 实现。主要负责登录、首页态势展示、监测点管理、图层地图、预警中心、报告上报和分析识别等交互界面。

2. 后端业务层

   后端使用 Fastify + TypeScript 实现 REST API。它负责用户登录鉴权、角色权限、业务数据读写、文件上传、AI 分析任务转发、地灾图层读取、审计日志、健康检查、运行期迁移和领域事件记录。

3. 数据与空间层

   当前本地演示默认使用 SQLite 存储业务数据，包括用户、监测点、预警、报告、分析记录、审计日志等。空间地灾数据保存在 `data/geohazards` 目录，包含 NASA/USGS 等公开数据源的 CSV、GeoJSON 和 GeoTIFF 文件。系统也预留 PostgreSQL/PostGIS 主库接入能力，用于后续更规范地管理空间数据。

4. AI 模型服务层

   模型服务使用 FastAPI + PyTorch，默认接口包括 `GET /health` 和 `POST /predict/landslide`。主后端通过 `AI_INFERENCE_BASE_URL` 调用模型服务，前端不直接访问模型服务。当前模型服务代码约定加载 `landslide_clf.pth` 和 `landslide_seg.pth`，用于滑坡分类和分割。

## 三、核心业务流程

1. 用户登录

   用户通过前端登录页提交账号密码。后端校验用户后签发 JWT，前端后续请求在请求头中携带 token。系统已区分 demo 和 production 模式，生产模式会阻止使用默认演示账号和弱 JWT 密钥。

2. 监测态势展示

   前端首页调用后端统计接口，展示监测点数量、预警态势、传感器状态、趋势图和系统运行状态。图表由 ECharts 渲染，地图由 Leaflet 展示。

3. 地灾图层展示

   前端选择图层后请求后端地灾接口。后端优先可从 PostGIS 获取空间数据，也可从本地 GeoJSON/CSV/GeoTIFF 数据包读取，再返回给前端地图组件展示。

4. 群众上报

   群众或值守人员提交灾情报告、位置、描述和图片。后端保存报告与附件，并记录审计日志和领域事件，便于后续流转、通知或大屏联动。

5. AI 图片识别

   用户上传滑坡相关图片后，后端先保存文件，再根据运行配置选择推理路径：

   - 如果配置了 `AI_INFERENCE_BASE_URL`，后端会把图片转发给外部模型服务，并规范化返回结果。
   - 如果未配置，demo 模式下会使用内置 mock 推理，生成稳定的演示结果。
   - production 模式必须配置真实模型服务地址，避免正式数据误写入 mock 结果。

   识别结果会保存到 `analysis_runs`，包括模型来源、置信度、摘要、分类结果和分割区域。

## 四、当前 YOLO 模型与数据状态

当前项目中已经存在 YOLO 相关数据和训练产物，但还没有接入网站的真实推理流程。

已具备内容：

- YOLO 数据集目录：`data/geohazards/yolo_landslide`
- 当前图像数量：训练集 78 张，验证集 12 张，共 90 张
- 数据配置文件：`data/geohazards/yolo_landslide/data.yaml`
- 训练权重：`models/yolo_landslide_from_masks/yolov8n_640/weights/best.pt` 和 `last.pt`
- 根目录还有 `yolov8n.pt`、`yolo26n.pt`

尚未完成的接入：

- 当前模型服务未安装和加载 Ultralytics YOLO。
- 当前 `/predict/landslide` 代码加载的是 ResNet18 分类器和 UNet 分割器，不是 YOLO `.pt` 权重。
- 后端分析接口当前运行状态下未配置 `AI_INFERENCE_BASE_URL`，因此本地网站上传分析时仍会走 mock 推理。
- SQLite 中 YOLO 数据集登记表目前为空，数据集尚未进入平台侧数据资产管理流程。

因此，准确结论是：YOLO 模型和数据已经在项目文件中准备好，但还没有真正接入 GeoProphet 网站的在线识别功能。

## 五、平台优点

1. MVP 闭环完整

   当前系统已经覆盖地灾监测平台的主要演示链路，包括登录、态势总览、地图可视化、监测点、预警、上报、AI 初审、审计和接口文档，适合进行业务汇报和需求确认。

2. 架构清晰，便于扩展

   前端、后端、模型服务相互分离。前端专注交互，后端专注业务和数据，模型服务专注推理。后续替换模型、迁移数据库或部署到容器环境时，改造边界比较清楚。

3. 已具备生产化意识

   系统已经加入 demo/production 模式、安全密钥校验、CORS 白名单、上传文件校验、限流、审计日志、requestId、健康检查和运行期迁移等机制。这些设计能降低从演示系统走向生产系统时的风险。

4. 空间数据基础较好

   项目已经沉淀 NASA/USGS 相关地灾数据包，并预留 PostGIS 接入能力。对于地质灾害平台来说，空间数据和地图图层是核心能力，当前底座方向正确。

5. AI 能力解耦合理

   主业务后端不直接绑定某个模型框架，而是通过 HTTP 调用模型服务。这种方式有利于后续接入 YOLO、分割模型、多模态模型或云端推理服务。

6. 本地演示启动简单

   使用 `npm run dev` 即可启动前后端开发环境，Docker Compose 也提供了后端、前端、模型服务和 PostGIS 的组合启动方式，方便联调和演示。

## 六、平台缺点与风险

1. YOLO 尚未真正接入业务

   虽然本地有 YOLO 数据集和训练权重，但网站当前分析功能没有调用 YOLO。演示时如果说“已接入 YOLO 在线识别”，会与实际状态不一致。更准确的表述应是“YOLO 训练数据和权重已准备，待接入模型服务”。

2. 数据集规模较小

   当前 YOLO 数据集共 90 张图像，规模偏小，且属于启动型/演示型数据。用于现场真实灾害识别时，泛化能力、误报率和漏报率都需要进一步验证。

3. 模型服务仍偏原型

   当前模型服务约定 `.pth` 分类和分割权重，缺权重时可在 demo 模式降级。正式环境需要明确模型版本、输入规范、输出格式、性能指标、推理耗时、GPU/CPU 部署方式和异常处理策略。

4. AI 分析是同步 HTTP 流程

   上传图片后后端同步等待模型服务返回。如果模型耗时较长、并发较高或模型服务不稳定，容易造成请求超时和用户等待。更适合改造成异步任务队列模式。

5. SQLite 更适合演示，不适合作为主生产库

   SQLite 启动简单，但在多人并发、复杂空间查询、备份恢复、权限隔离和高可用方面不适合作为正式主库。生产建议迁移到 PostgreSQL/PostGIS。

6. 真实监测数据接入不足

   当前平台主要是演示数据和静态数据包。若要用于真实地灾监测，需要接入传感器、雨量、GNSS、裂缝计、倾角计、遥感影像、巡查记录等实时或准实时数据源。

7. 业务闭环还不够深

   当前有预警、上报和分析展示，但事件处置流程、工单流转、人员通知、复核确认、应急响应、闭环归档等生产业务能力仍需补齐。

8. 运维观测能力仍需增强

   系统已有健康检查和 requestId，但还缺少完整的指标监控、集中日志、链路追踪、错误告警、模型性能监控和数据质量监控。

9. 权限体系仍需完善

   已有基础 RBAC 和审计，但还缺账号管理页面、密码策略、账号禁用、密码重置、角色管理、组织机构、多租户和 OIDC 实际接入。

## 七、改进建议

1. 优先完成 YOLO 在线接入

   在 `model-service` 中安装 `ultralytics`，加载 `models/yolo_landslide_from_masks/yolov8n_640/weights/best.pt`，将 YOLO 检测框转换为平台统一的 `classification` 和 `segmentation.regions` 输出。然后启动模型服务，并让后端配置 `AI_INFERENCE_BASE_URL=http://127.0.0.1:8000`。

2. 建立模型版本管理

   为每次训练记录数据集版本、权重路径、训练参数、评估指标、混淆矩阵、样本范围和上线状态。前端模型库不应只展示静态演示数据，而应能反映真实模型资产。

3. 扩大并清洗训练数据

   当前 90 张图像只能作为初始验证。后续应增加更多区域、季节、尺度、传感器来源和负样本，并引入专家复核标注，降低误报和漏报。

4. 将 AI 分析改为异步任务

   采用任务表、队列或 outbox worker。用户提交后立即返回任务 ID，前端轮询或 WebSocket 获取进度，模型服务后台处理。这样更适合大图、遥感影像和多模型流水线。

5. 推进 PostGIS 主库落地

   将空间图层、监测点、灾害事件、影响范围和风险区划统一纳入 PostGIS，提升空间查询、叠加分析和数据治理能力。

6. 补齐生产运维能力

   增加日志聚合、指标监控、错误告警、备份恢复、部署脚本、灰度发布和模型服务健康监控。生产模式下应关闭所有 mock/fallback。

7. 补齐处置业务闭环

   围绕“发现风险、模型初判、人工复核、发布预警、处置派单、结果反馈、归档复盘”设计完整流程，让平台从展示型 MVP 走向业务系统。

## 八、总体结论

GeoProphet 当前是一个结构清晰、演示闭环完整、具备生产化雏形的地灾监测平台 MVP。它的优势在于前后端分离、地灾数据基础较完整、AI 服务解耦、具备审计和安全加固方向；主要短板在于真实模型尚未接入、YOLO 仍停留在文件和训练产物阶段、数据集规模较小、实时数据和生产处置闭环不足。

如果下一步目标是比赛展示或项目汇报，当前版本已经可以支撑“平台能力演示”。如果目标是强调 AI 识别能力，应优先完成 YOLO 推理服务接入，并在演示口径中明确模型数据、训练结果、在线推理链路和当前准确率边界。
