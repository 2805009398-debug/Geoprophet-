# 真实业务逻辑缺陷分析

分析日期：2026-05-25。  
分析对象：当前 GeoProphet Web MVP 的前端、后端、模型服务、数据库初始化脚本和部署配置。本文只讨论真实业务逻辑缺陷，不把代码风格、UI 美观、依赖版本这类纯技术项混在一起。

## 总体判断

当前项目更接近“演示型 MVP”：可以登录、查看演示点位、展示预警、提交群众上报、上传图片并调用演示/模型推理。但从地灾监测业务闭环看，它还没有形成“数据接入 -> 质控入库 -> 风险研判 -> 预警发布 -> 群众上报复核 -> 应急处置 -> 归档审计”的完整业务链路。

下面列出的缺陷都能在代码中找到明确证据，不是推测。

## P0：上线前必须处理

### 1. 数据接收业务没有真实入口

证据：

- `backend/src/routes` 中没有传感器观测值、视频事件、遥感成果、网关心跳等数据接入接口。
- `backend/db/init.sql:145-195` 只写入静态传感器和观测演示数据。
- `backend/src/routes/dashboard.ts:120-129` 的 `channelCount`、`avgLatencyMs`、`successRate` 是硬编码指标。
- `frontend/src/views/DataCenterView.vue:111-116` 的处理流程只是静态步骤展示。

业务影响：

- 系统不能接收真实监测设备数据，无法作为“地灾监测数据接收与管理平台”使用。
- 接入通道、成功率、平均响应时间不是运行数据，汇报时容易被误认为真实监测能力。
- 后续预警、趋势、评估都只能基于初始化脚本，无法随业务运行更新。

建议完善：

- 增加 `POST /api/ingestion/observations`、`POST /api/ingestion/device-heartbeats`、`POST /api/ingestion/remote-sensing-products`。
- 增加数据批次、原始报文、解析状态、质控结果、错误原因表。
- Dashboard 的接入指标从批次表和日志表聚合，不再写死。

验收标准：

- 能模拟设备批量上报观测值。
- 异常格式、重复数据、越界数值能进入错误队列。
- 接入中心指标随真实请求变化。

### 2. 预警业务只有“新增/确认”，没有完整处置闭环

证据：

- `backend/src/routes/alerts.ts:54-69` 只能新增 active 预警。
- `backend/src/routes/alerts.ts:71-82` 只能把 active 更新为 acknowledged。
- `backend/db/init.sql:50-62` 的 `alerts` 表没有责任人、处置状态、关闭时间、升级记录、会商意见、发布范围等字段。

业务影响：

- 真实业务中，“确认”只代表有人看到了，不代表预警已处置。
- 不能表达误报、升级、降级、解除、转交、超时、发布、撤离、归档。
- 无法追踪谁在何时做了什么处置动作，不适合值班和应急责任追溯。

建议完善：

- 增加预警状态机：`draft -> active -> acknowledged -> processing -> resolved -> closed`，并支持 `cancelled/false_positive/escalated`。
- 增加预警事件流水表 `alert_events`，记录确认、指派、会商、升级、解除、关闭。
- 区分“内部研判预警”和“对外发布预警”。

验收标准：

- 每次预警状态变化都能查到操作人、时间、原因和备注。
- 支持关闭预警时填写处置结果。
- 支持从高置信 AI 识别或群众上报自动生成待审核预警草稿。

### 3. 群众上报没有真实审核流程，AI 初审置信度是随机数

证据：

- `backend/src/routes/reports.ts:48-70` 新增上报后直接进入 `pending`。
- `backend/src/routes/reports.ts:55` 置信度是 `0.55 + Math.random() * 0.35`。
- 没有“审核通过/驳回/转预警/合并重复线索/专家复核”的接口。
- `frontend/src/views/ReportsView.vue:170-174` 展示“AI 初审 -> 专家复核”流程，但没有对应业务动作。

业务影响：

- 置信度不代表图像或文本风险，只是随机展示数字。
- 群众上报不能被真正处理，无法变成预警、研判任务或处置工单。
- 运营人员无法区分无效上报、重复上报、高危线索。

建议完善：

- 提交上报后触发真实 AI 初审任务，保存模型版本、输入图片、输出标签、置信度和建议。
- 增加审核接口：`PATCH /api/reports/:id/review`。
- 支持将上报转为预警草稿或关联已有预警。

验收标准：

- 同一张图片重复提交，AI 初审结果稳定可复核。
- 审核动作有操作记录。
- 高置信上报可以一键转入预警中心。

### 4. AI 识别结果没有进入地灾业务闭环

证据：

- `backend/src/services/hazard-analysis.ts:422-452` 只把识别结果写入 `analysis_runs`。
- `analysis_runs` 与 `sites`、`crowd_reports`、`alerts`、`assessments` 没有外键关系。
- `frontend/src/views/AnalysisView.vue:302-317` 只展示最近识别记录。

业务影响：

- 滑坡/冰川识别只是孤立工具，不能驱动点位风险等级、预警生成或影响评估。
- 业务人员无法知道某次模型识别对应哪个监测点、哪条群众上报、哪次预警。
- 模型结果不能被复核、采纳、驳回或归档到处置链。

建议完善：

- 给 `analysis_runs` 增加 `site_id`、`report_id`、`alert_id`、`review_status`、`reviewer_id`。
- 识别完成后生成“研判建议”，由专家确认后影响预警或评估。
- 保存模型版本、阈值、输入文件 hash，保证结果可追溯。

验收标准：

- 从任意点位详情能看到相关识别记录。
- 从群众上报能发起识别，识别结果能回写上报审核意见。
- 专家采纳模型结果后能生成预警或影响评估。

### 5. 权限角色存在，但业务权限不存在

证据：

- `backend/src/db.ts:7-11` 有 `admin/operator/expert` 三类角色。
- `backend/src/guards.ts:3-8` 只校验是否登录。
- 所有需要登录的路由都只使用 `authenticate`，没有按角色授权。
- `frontend/src/components/AppShell.vue:72-75` 只展示角色，不控制功能。

业务影响：

- 值班员、专家、管理员在系统内权限相同。
- 任何登录用户都能新增预警、确认预警、查看所有群众手机号和上传图片。
- 不符合政府/应急业务中的最小权限、岗位分权和责任边界。

建议完善：

- 定义角色权限矩阵：管理员、值班员、专家、公众、审计员。
- 后端增加 `requireRole` 或 `requirePermission`。
- 前端菜单和按钮按权限显示，但最终以后端授权为准。

验收标准：

- operator 不能管理用户和系统配置。
- expert 可以复核研判，但不能随意创建系统账号。
- 未授权访问返回 403，并写入安全日志。

## P1：影响业务可信度和交付质量

### 6. 时间数据固定在演示日期

证据：

- `backend/db/init.sql:137-240` 演示数据集中在 2026-04-19 到 2026-04-23。
- `backend/src/routes/dashboard.ts:5` 趋势结束日期固定为 `2026-04-24T00:00:00`。

业务影响：

- 到 2026-05-25 运行时，首页仍展示 2026-04 的趋势，不能反映当前态势。
- 演示稳定，但真实值守会误导用户。

建议完善：

- 趋势结束日期用“最新业务数据日期”或“当前日期”，并支持演示模式配置。
- 数据库种子数据和生产数据分离。

验收标准：

- 有真实数据时按真实最新日期展示。
- 无数据时明确显示“暂无数据”，不伪造趋势。

### 7. 风险等级、灾害类型和状态口径不统一

证据：

- `backend/db/init.sql:137-143` 点位灾害类型包含“滑塌”。
- `backend/src/routes/reports.ts:14` 群众上报类型只允许“滑坡、泥石流、崩塌、沉陷、裂缝”，不含“滑塌”。
- `sites.status` 出现 `warning/alert/watch/active/normal`，`alerts.status` 出现 `active/acknowledged`，`crowd_reports.status` 出现 `pending/verified/reviewing`，没有统一字典表。
- 前端多处直接展示英文状态，例如 `frontend/src/views/SitesView.vue:152-160`。

业务影响：

- 统计口径容易混乱，同一类灾害在点位和上报中可能无法对应。
- 后续报表、筛选、模型标签映射会出现漏算。
- 用户看到中英文混杂状态，业务含义不清。

建议完善：

- 建立灾害类型、风险等级、点位状态、预警状态、上报状态字典。
- 前后端统一使用 code + label，不在页面直接展示原始 code。
- 明确“滑坡”和“滑塌”的归类关系。

验收标准：

- 所有筛选项来自字典接口。
- 报表统计基于统一 code。
- 页面展示中文业务标签，接口仍保留稳定 code。

### 8. 点位空间业务规则缺失

证据：

- `backend/src/routes/reports.ts:17-18` 只要求经纬度是数字。
- `backend/db/init.sql:13-25` 点位只有经纬度，没有行政区边界、影响范围、缓冲区。
- 群众上报选择点位后只是前端同步经纬度：`frontend/src/views/ReportsView.vue:39-48`。

业务影响：

- 可以提交任意经纬度，包括白山市外或明显错误位置。
- 不能自动判断上报是否靠近某个隐患点。
- 无法做“影响范围内人口/道路/村屯”的空间评估。

建议完善：

- 增加行政区边界、点位影响半径、风险区 polygon。
- 上报时根据坐标自动匹配附近点位和行政区。
- 重要评估迁移到 PostGIS 或等价空间能力。

验收标准：

- 白山市外坐标被提示或进入异常队列。
- 未选择点位时系统能推荐最近隐患点。
- 影响评估能基于空间范围计算。

### 9. 影响评估只是静态展示

证据：

- `backend/db/init.sql:93-103` 定义 `assessments`。
- `backend/db/init.sql:219-223` 只插入静态评估数据。
- `backend/src/routes/analysis.ts:18-38` 只提供查询，没有创建、更新、复核或重新计算接口。

业务影响：

- 受影响人口、经济损失、道路影响不能随预警、降雨、点位风险变化更新。
- 评估结果缺少来源、计算方法、版本和复核人。

建议完善：

- 增加评估任务表和评估结果版本表。
- 记录输入数据、计算模型、参数、人工修正和审核状态。
- 支持从预警或点位发起影响评估。

验收标准：

- 同一点位可保留多版评估。
- 每版评估都有来源和审核状态。
- 已发布评估不可被静默覆盖。

### 10. 应急处置闭环能力已从当前版本取消

证据：

- 当前版本已取消预案页面和 `/api/plans` 接口。
- `backend/db/init.sql` 不再初始化预案表和预案示例数据。
- 前端导航和路由不再暴露预案入口。

业务影响：

- 当前版本不提供预案模板、启动实例、任务派发和资源调度。
- 预警处置仍依赖推荐动作、人工联动和线下流程。

建议完善：

- 如果后续恢复该能力，再拆分预案模板、启动实例、任务、资源、联系人。
- 明确触发条件、审批流程、执行回执和审计要求后再进入开发。

验收标准：

- 当前版本访问 `/plans` 不再进入预案页面。
- API 文档不再包含 `/api/plans`。
- 用户界面不再出现应急预案功能入口。

### 11. 系统日志不是业务审计日志

证据：

- `backend/db/init.sql:129-135` `system_logs` 只有 category、level、message、created_at。
- `backend/db/init.sql:230-240` 日志全是初始化数据。
- 登录、新增预警、确认预警、提交上报、上传文件、模型识别都没有写审计日志。

业务影响：

- 无法追溯关键业务操作。
- 发生误报、漏报、恶意上传或误操作时，没有责任链。

建议完善：

- 增加 `audit_logs`，记录 actor、action、entity_type、entity_id、before、after、ip、user_agent。
- 关键业务动作统一写审计。

验收标准：

- 每次预警状态变化都能在审计日志中查询。
- 上传和模型识别能追踪到操作者和源文件。

## P2：影响长期演进

### 12. 公众入口和内部系统边界不清

证据：

- `POST /api/reports` 和 `POST /api/uploads` 不需要登录。
- 同一个 API 服务同时承载公众上报、内部预警、管理看板。
- 上传文件直接通过 `/uploads/` 公开访问。

业务影响：

- 公众入口容易被滥用，内部系统也暴露在同一边界上。
- 群众手机号、上报内容和图片属于敏感数据，需要更严格的数据分级。

建议完善：

- 区分公众 API 和内网管理 API。
- 公众上报增加验证码、限流、内容审核和隐私脱敏。
- 上传文件用授权访问或临时签名 URL。

### 13. 需求覆盖文案与真实能力存在落差

证据：

- `README.md:5-15` 写到“数据接收与管理、AI 初审、系统集成”等能力。
- `docs/requirement-summary.md:3-16` 写到“接收状态指标、异常日志、算法模型库、影响评估”等覆盖。
- 实际代码中多项能力是静态数据、占位配置或演示逻辑。

业务影响：

- 交付验收时容易被追问“真实数据在哪里接入、审核在哪里完成、模型结果如何影响预警”。
- 项目陈述容易超出当前可验证功能。

建议完善：

- 文档区分“已实现”“演示占位”“待接入”。
- 每个需求条目绑定接口、页面、数据表和验收方式。

### 14. 模型服务业务语义不完整

证据：

- `model-service/app/services/landslide.py:128-155` 和 `model-service/app/services/glacier.py:69-90` fallback 会返回演示推理。
- 后端 `backend/src/services/hazard-analysis.ts:208-276` 只归一化结果，没有保存模型输入条件、影像时间、空间范围、人工复核结论。

业务影响：

- 推理结果难以进入正式专家研判。
- 对遥感/InSAR 数据来说，只上传一张图片缺少时间、轨道、区域、分辨率等关键元数据。

建议完善：

- 识别任务表增加业务元数据：采集时间、空间范围、传感器类型、影像批次、模型版本、阈值。
- fallback 结果在业务界面明确标记“演示，不可用于正式研判”。

## 推荐业务改造路线

1. 先补真实数据接入和错误队列，否则系统无法从演示进入运行。
2. 补预警状态机、群众上报审核、AI 结果复核，让业务闭环跑起来。
3. 补权限、审计、公众入口边界，满足安全和责任追溯。
4. 统一字典、空间规则和评估闭环，把静态展示升级为可运营平台。
